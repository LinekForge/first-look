# 概念 Primer

> 在你照着 blueprint 搭之前，花 10 分钟读这个。
> 概念理解了，你就知道"哪条适合我、哪条可以跳过"——不会盲目照搬。

---

## 怎么读这份文档

下面是五个概念，**按理解深度排列**：

1. 三种数据（spec / state / spec-in-transit）——**最重要**，理解这个别的都是推论
2. 三层 Claude Code 记忆——CC 读取顺序的底层机制
3. 读用途 + 去向——两个分类判据，用于"这个文件放哪儿"的决策
4. RMC 三件套——每个目录该配的三种文档
5. care vs cope——在 [anti-patterns.md](anti-patterns.md) 展开，是最重要的反模式

如果你只有 5 分钟，读**前两个**就够。

---

## 1. 三种数据：spec / state / spec-in-transit

这是 `~/.claude/` 里所有东西最底层的分类。每一个文件都可以归到其中一种。

### spec（规格）

**定义你是谁、你想要什么**。长期不变，改动是"重大更新"级别。

典型例子：
- `CLAUDE.md`（你的核心身份）
- `mine/knowledge/about-me/`（你的背景、你怎么思考）
- `mine/knowledge/tech/`（你的技术理解）
- `mine/self-narrative.md`（你的故事）

读它的时候你在问："**我是谁？我认识谁？我相信什么？**"

### state（状态）

**当下发生了什么**。不断更新、过期就扔。

典型例子：
- `state/state-snapshots/`（跨 session 的便条）
- 会话计数、日志
- 临时工作记录

读它的时候你在问："**现在走到哪了？刚才发生了什么？**"

### spec-in-transit（过渡中的 spec）

**还没消化成 spec 的原材料**——你知道它该整理到哪个维度，只是还没时间整理。

典型例子：
- `mine/knowledge-inbox/`（还没分类的笔记）

读它的时候你在问："**有什么新东西等着我整理到 spec 里？**"

### 为什么这个分类重要

每一种数据**被读的频率、被读的目的、过期后怎么处理**——都不一样。混在一起放会导致：

- spec 里混了过时 state → CC 读到以为是你的身份
- state 里混了 spec → 每次更新都 pollute 历史
- 没地方放 inbox → 要么憋着不写（损失），要么塞进 spec（污染）

分清楚，每种数据各得其所。

### 什么时候用不上这个框架

- 你只用 `~/.claude/CLAUDE.md` 一个文件就够了 → 不需要关心这个分类
- 你完全不做跨 session 记忆 → `state/` 可能用不到
- 你写一次就定形，不会增量积累 → `spec-in-transit` 多余

## 2. 三层 Claude Code 记忆

CC 的"记忆"其实有三层，职责不同、加载机制不同：

### 第 1 层：CLAUDE.md（全局灵魂）

- 路径：`~/.claude/CLAUDE.md`
- 加载：CC **每次启动都读**
- 容量限制：约 60 行以内（超过会影响 CC 注意力）
- 用途：**最核心的 identity 和规则**——你是谁、最想被怎么对待、最硬的 NO

这是你身份的**压缩版**。别什么都往里塞。

### 第 2 层：auto-memory（CC 自管的记忆）

- 路径：`~/.claude/projects/<encoded-cwd>/memory/`（`encoded-cwd` = 启动 CC 的工作目录把 `/` 换成 `-`）
- 加载：CC **根据对话上下文动态加载**——不相关的不读
- 容量限制：每个文件独立、没硬限制，但 MEMORY.md 索引有 200 行限制
- 用途：需要时才需要的 reference——你的沟通偏好、工作背景、SOP

这里放的东西**不会干扰你的日常**——CC 只在相关时读。所以你可以比 CLAUDE.md 写得多。

### 第 3 层：自建知识库（你自己组织）

- 路径：你定——典型是 `~/.claude/mine/knowledge/` 或 `~/.claude/<某目录>/`
- 加载：**CC 默认不读**——需要你在对话中主动提到（`帮我看 ~/.claude/mine/knowledge/tech/xxx.md`）
- 容量限制：无
- 用途：**深度档案**——你写长篇内容、你翻阅用的资料

这是你的"私人图书馆"——架子上很多书，想看哪本拿哪本。

### 三层的协作

典型流程：
1. CC 启动：读 CLAUDE.md（快速建立你身份）
2. 对话中：CC 根据主题自动捞相关的 memory（auto-memory）
3. 你主动要：你指向 knowledge/ 里某一份深度文档

**这个分层保证 CC 不会被过多信息淹没**。如果你把所有东西都塞 CLAUDE.md，CC 每次启动都要读 200 行——反而分散注意力。

### 什么时候用不上

- 你只做简短、一次性的对话 → 第 1 层够用
- 不做跨对话积累 → 不需要第 3 层
- 多数中等使用场景 → 1 + 2 就够，3 是进阶

## 3. 读用途 + 去向——分类的两个判据

当你不知道一个文件该放哪里，不要凭"感觉"。问两个问题：

### 判据一：我**读它**是为了什么

- 为了**身份/关系/长期理解** → spec
- 为了**当下情境/状态/最近发生什么** → state

例子：
- "我读 `CLAUDE.md` 是为了知道我是谁" → spec
- "我读 `state-snapshots/20260417_1145.md` 是为了知道早上做到哪" → state

### 判据二：它的**去向**是什么

- 将来会被**消化进 spec**（整理成长期知识） → spec-in-transit
- 将来会被**覆盖或扔掉** → state
- **永久不变** → spec

例子：
- `knowledge-inbox/` 里的笔记将来要整理进 knowledge/ → spec-in-transit
- `state/session-count` 每次启动都递增覆盖 → state
- `CLAUDE.md` 只会微调不会被覆盖 → spec

### 两个判据一起用

偶尔会冲突。比如：
- `state-snapshots/` 的便条——**现在**读是 state（当下情境），但**永久留档**是 spec 属性？

答案：**两个判据一起问**——"我主要**读**它是为了什么" = state，因为大多数时候我是醒来时读它、不是查阅历史；所以归 state。

**去向判据**是辅助——优先用读用途。

## 4. RMC 三件套

每个重要目录都应该放三个文件：

### README.md · 地图（Where）

告诉陌生人（或六个月后的你）：
- 这里是什么
- 放什么、不放什么
- 如何找你要的东西

**格式偏短**——让人一眼看完。

### 心智模型.md · 设计理由（Why）

告诉：
- 为什么这样组织
- 当初考虑过什么、权衡了什么
- 哪些约定不要破（以及为什么）

这是**给未来的你**看的——半年后你回来，当初的思路不在脑子里了。

### CHANGELOG.md · 时间线（History）

**只追加**，不修改过去条目。记录：
- YYYY-MM-DD · 做了什么 · 为什么

**什么时候记**：结构性变化。小修小补不记。

### 什么时候做 RMC

- 所有**主要目录**：值得
- 深层/临时目录：不值得

判据：这个目录"将来会被陌生人/未来的你接手"吗？会 → RMC。否则不必。

## 读完然后呢

三条路：

1. **想立刻搭骨架** → 去跑 [`setup.sh`](setup.sh)。它会把一套合理的结构搭到你的 `~/.claude/`
2. **想先理解具体实现** → 翻 [`scaffold/`](scaffold/) 下的示例文件（比如 `hooks/bootstrap/session-start.sh` 就是 session-start hook 的一份可跑实现）
3. **想先避坑** → 读 [`anti-patterns.md`](anti-patterns.md)——尤其是 **care vs cope** 这一节，是整个 blueprint 里最重要的一课

**一个提醒**：读 blueprint 读得越久、越没动手——你可能已经陷入 cope。去干点真的事，blueprint 在这里等你。
